# Lab 5: Agile and Scrum Practice

Данная лабораторная работа посвящена изучению и применению методологий гибкой разработки (Agile) с использованием фреймворка Scrum. В рамках работы реализовано управление проектом через инструменты GitHub: Issues, Projects и Milestones.

---

## 1. Product Backlog (User Stories)

В бэклог включены 6 ключевых пользовательских историй, каждая из которых содержит критерии приемки (Definition of Done).

1.  **Интерфейс списка задач** — Отображение всех текущих дел пользователя.
2.  **Отметка задачи как выполненной** — Визуальное подтверждение завершения дела.
3.  **Удаление задач** — Очистка списка от ненужных элементов.
4.  **Приоритизация (Labels)** — Выделение важных задач цветовыми метками.
5.  **Установка дедлайнов** — Контроль сроков выполнения.
6.  **Поиск по списку** — Быстрая навигация по ключевым словам.

---

## 2. Sprint Planning (Sprint 1)

**Цель спринта:** Создать минимально жизнеспособный продукт (MVP) персонального менеджера задач.
**Срок:** 1 неделя.

Задачи распределены по Канбан-доске в колонках:
* **Todo:** Задачи в очереди на выполнение.
* **In Progress:** Задачи, находящиеся в активной разработке.
* **Done:** Завершенные задачи, соответствующие критериям Definition of Done.

---

## 3. Sprint Review (Обзор спринта)

В ходе инспекции результатов спринта было установлено:
* **Реализовано:** Полностью готовы интерфейс и логика отметки выполненных задач (Issues #1, #2).
* **В процессе:** Функционал удаления и приоритизации находится на стадии тестирования (Issues #3, #4).
* **Демонстрация:** Продукт позволяет пользователю видеть список дел и менять их статус, что соответствует изначальной цели MVP.

---

## 4. Sprint Retrospective (Ретроспектива)

Анализ процесса работы команды:

### Что прошло хорошо?
- Использование GitHub Issues позволило четко зафиксировать Definition of Done, что исключило двусмысленность при приемке задач.
- Автоматизация перемещения карточек на Project Board упростила мониторинг прогресса.

### Что можно улучшить?
- Оценка сложности некоторых задач (например, поиск) оказалась заниженной. В следующем спринте стоит использовать Story Points.
- Необходимо чаще синхронизировать изменения в README с актуальным статусом на доске.

### План действий:
1. Внедрить Planning Poker для более точной оценки времени.
2. Настроить GitHub Actions для автоматического закрытия Issues при слиянии Pull Request.
